# R+4 Trusted-Setup Ceremony — Coordinator Runbook

**Spec:** [dcslabs.ai/standard/r4](https://dcslabs.ai/standard/r4) §6 Setup Ceremony
**Purpose:** produce the *production* Groth16 proving/verifying keys for the
`r4-threshold-count-v1` circuit in a way no single party can subvert.

> **Why this exists and cannot be skipped.** A Groth16 setup produces secret
> randomness ("toxic waste"). Whoever knows it can forge proofs for *any*
> statement. A multi-party ceremony has each contributor add their own secret
> and destroy it. As long as **one** contributor was honest, the toxic waste
> is unrecoverable and the keys are trustless. This is not a code step — it is
> a coordinated human event, by design. The R+4 spec mandates **≥5 independent
> contributors over a ≥14-day window.**

This is the one part of R+4 that is genuinely an operational event, not
software. Everything else (circuit, build pipeline, prover, verifier) is
code-complete.

---

## Phase 1 — universal Powers of Tau

**Do not run a fresh Phase 1.** Reuse an existing perpetual Powers-of-Tau —
already the product of hundreds of contributors. The `threshold-count` circuit
compiles to **1,197,221 R1CS constraints** (`snarkjs r1cs info`), so Phase 2
setup needs a **power-22** ptau (2^22 ≈ 4.2M ≥ 2·constraints — power 16 is far
too small). Use the Hermez `powersOfTau28_hez_final_22` file, verify it, and
place it at `ceremony/pot22_final.ptau`.

```
snarkjs powersoftau verify pot22_final.ptau     # must print "Powers of Tau OK"
```

## Phase 2 — circuit-specific ceremony

Phase 2 is specific to the threshold-count circuit. This is the part DCS Labs
coordinates.

### Coordinator steps

1. **Initialise** from the verified Phase 1:
   ```
   snarkjs groth16 setup threshold-count.r1cs pot22_final.ptau phase2_0000.zkey
   ```
2. **Open the contribution window** — announce publicly (≥14 days). Publish:
   the `.r1cs`, the circuit source, this runbook, and `phase2_0000.zkey`.
3. **Collect contributions** — each contributor runs `contribute.sh` (below)
   on their own machine with fresh entropy and sends back their `.zkey` plus
   their attestation. Minimum 5 independent contributors.
4. **Chain the contributions** — apply them in order; each one's hash must
   match the contributor's published attestation.
5. **Apply a random beacon** — finalise with a public, unpredictable value
   (e.g. a future Bitcoin block hash) so the last word isn't the coordinator's:
   ```
   snarkjs zkey beacon phase2_N.zkey phase2_final.zkey <beacon_hash> 10
   ```
6. **Verify the full chain**:
   ```
   snarkjs zkey verify threshold-count.r1cs pot22_final.ptau phase2_final.zkey
   ```
   Must print `ZKey Ok`.
7. **Publish** — `phase2_final.zkey`, `verification_key.json`, the full
   transcript, and every contributor attestation. Register the `vk_hash` in
   the R+4 registry and anchor it on Base mainnet (§6.5).
8. Place `phase2_final.zkey` and `pot22_final.ptau` in `ceremony/`, then run
   `./build.sh prod`.

### Contributor steps

Each of the ≥5 contributors:

1. Receives the previous `.zkey` (start: `phase2_0000.zkey`).
2. Runs `./contribute.sh <prev.zkey> <my-name>` on an **air-gapped machine**
   with a fresh OS install and a hardware entropy source if possible.
3. Sends back the produced `.zkey` and the printed contribution hash.
4. **Destroys the machine's entropy / wipes the disk.** The whole security
   model rests on this — the contributor's secret must not survive.

A contributor needs only a GitHub account and a clean machine. The more
mutually-independent the contributors, the stronger the guarantee.
