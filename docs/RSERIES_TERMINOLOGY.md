# R-Series Terminology Standardization

**Project:** DCS Labs R-Series AI-Agent Trust Stack
**Document type:** Controlled vocabulary / definitions reference
**Status:** Authoritative — all other R-Series documents must conform to this glossary
**Last updated:** 2026-05-23

---

## 1. Purpose & How to Use This Glossary

An external reviewer observed that R-Series documentation uses maturity and
deployment words — *prototype*, *operational*, *production*, *live*, *verified*,
*mainnet*, *federated*, *dev keys* — loosely and interchangeably. Used loosely,
these words overclaim: they imply audits, adversarial testing, or multi-party
guarantees that do not yet exist.

This document fixes a single, precise meaning for each term. **The goal is
anti-overclaiming.** Every claim about an R-Series layer must be supported by the
specific evidence listed against the term used. If the evidence does not exist,
the term may not be used — pick a weaker, accurate term instead.

**How to use it:**

1. When writing any R-Series document, use only the terms defined in Section 2.
2. Before applying a maturity term to a layer, confirm the required evidence
   exists. If it does not, downgrade the claim.
3. For per-layer status language, copy the exact approved phrase from Section 3.
   Do not paraphrase it.
4. When in doubt, claim *less*. An accurate "operational prototype" is stronger
   than a contestable "production system".

## 2. Definitions

| Term | Precise meaning | Evidence required to claim it |
|---|---|---|
| **Prototype** | A component that is functionally working *in isolation*. Its core logic executes and produces correct output for hand-picked inputs, but it has not been exercised end-to-end as part of the full stack, nor tested reproducibly. | Runnable code; a demonstration of the core function succeeding on at least one representative input. |
| **Operational prototype** | A component that works **end-to-end** within the real stack and has been **reproducibly tested** — the same test suite can be re-run by another engineer and yields the same pass result. It is *not* independently audited and *not* adversarially tested. This is the highest maturity level currently claimable by any R-Series layer. | An end-to-end test run; a documented, re-runnable test procedure; recorded conformance results. |
| **Production** | A component that is (a) **independently audited** by a party outside DCS Labs, (b) **adversarially tested** (deliberate attempts to break it, e.g. malformed inputs, malicious provers, replay), and (c) **resilient** under failure and load. **No R-Series layer (R+2, R+3, R+4) meets this definition today.** The word "production" must not be applied to any layer. (The narrower phrase "production keys" is defined separately below and is permitted.) | A third-party audit report; a documented adversarial test campaign; resilience/failure-mode testing results. |
| **Verified** | A *specific named check* was executed and returned the *expected* result. "Verified" is never a general status — it always names the check. Example: "the Groth16 proof verified" means the on-chain verifier contract's `verifyProof` call returned `true` for that proof. | The name of the check, the input it ran against, and the recorded expected-vs-actual result. |
| **On-chain / anchored** | A real, confirmed transaction exists on a public blockchain and can be inspected by anyone via a block explorer. "Anchored" specifically means a hash/commitment was written into such a transaction. | A transaction hash on a public chain; the chain name; an explorer link. |
| **Testnet vs Mainnet** | **Testnet** = Base Sepolia: a public test network using valueless test tokens; suitable for integration testing, not for trust claims. **Mainnet** = Base: the public production blockchain. State the chain explicitly; never say only "on-chain" when the distinction matters. | The exact chain name (Base Sepolia or Base) accompanying every transaction reference. |
| **Dev keys vs Production keys** | **Dev keys** are cryptographic parameters generated by a **single party** in one place — convenient but offering no guarantee that toxic waste was destroyed. **Production keys** are the output of a **multi-party ceremony** (see *Ceremony-backed*), where security holds if at least one participant was honest. | For production keys: a ceremony transcript identifying multiple distinct contributors. |
| **Ceremony-backed** | Keys derived from a **multi-party trusted setup ceremony**, where the final parameters are secure as long as one contributor honestly discarded their secret. **v0.1 caveat:** the R+4 v0.1 ceremony is complete, but contributor independence has not been externally established — contributors are not yet confirmed to be mutually independent parties. The phrase "ceremony-backed" must always carry this v0.1 caveat until independence is verified. | A published ceremony transcript; contributor list; the explicit v0.1 independence caveat. |
| **Federated** | **Multiple independent issuers or nodes**, operated by distinct parties, verifying each other's claims. A single-operator, single-domain deployment is **not** federated regardless of architecture. **R+4 is not federated:** multi-issuer federation is specified but not built — it is single-domain. | Two or more independently operated nodes/issuers in live mutual verification. |
| **Conformance** | A component passes the **canonical R-Series test vectors** — the agreed, fixed set of input/output pairs that define correct behavior. A conformance score (e.g. "10/10") reports how many canonical vectors passed. Conformance is *not* a security claim; it measures correctness against the spec, not resistance to attack. | The canonical test-vector set; a recorded pass count out of total. |

## 3. Correct Per-Layer Usage

| Layer | Single approved status phrase | Phrases **not** to use |
|---|---|---|
| **R+2 — Open Provenance Standard** | "Operational prototype: end-to-end tested, conformance 10/10, with the on-chain identity contract live on Base mainnet." | "production", "production-ready", "fully verified", "audited". (Note: "on-chain identity contract live on Base mainnet" *is* accurate and may be stated.) |
| **R+3 — Tamper-Evident Audit Export** | "Operational prototype: end-to-end tested, with on-chain anchors on both Base Sepolia and Base mainnet." | "production", "audited", "tamper-proof" (use "tamper-evident"). Do not imply mainnet-only — anchors exist on both testnet and mainnet. |
| **R+4 — Federated ZK Verification** | "Cryptographic prototype with production keys: real circuit, completed v0.1 trusted-setup ceremony, and a Groth16 verifier deployed to Base mainnet. Multi-issuer federation is specified only." | "federated" (it is single-domain), "production" (the *system* is not — only the keys are ceremony-backed), "live federation", "multi-issuer" as a present-tense capability. The name "Federated ZK Verification" describes the *target design*, not current status. |

## 4. Worked Examples — Wrong vs Right

**Example 1 — R+4 maturity.**
*Wrong:* "R+4 is a production-grade federated zero-knowledge verification system."
*Right:* "R+4 is a cryptographic prototype with production keys; its Groth16 verifier
is deployed to Base mainnet. Federation is specified but not yet built, so the
current deployment is single-domain."
The wrong version claims independent audit ("production-grade") and multi-party
operation ("federated") — neither exists.

**Example 2 — keys vs system.**
*Wrong:* "R+4 uses production cryptography, so it's production-ready."
*Right:* "R+4 uses production keys — output of the completed v0.1 ceremony — but
the system as a whole is a prototype, since it has not been independently audited
or adversarially tested." Production *keys* do not make a production *system*; and
the v0.1 ceremony's contributor independence is not yet externally established.

**Example 3 — on-chain precision.**
*Wrong:* "R+3 anchors are verified on-chain."
*Right:* "R+3 writes anchor transactions to Base Sepolia and Base mainnet; each
anchor can be inspected via its transaction hash on a block explorer." Name the
chains and the evidence; do not let "verified" stand without naming the check.

**Example 4 — conformance vs security.**
*Wrong:* "R+2 passed conformance 10/10, so it is secure and verified."
*Right:* "R+2 passes all 10 canonical conformance vectors, confirming correctness
against the spec. Conformance is not a security result — R+2 has not been
adversarially tested or independently audited."
